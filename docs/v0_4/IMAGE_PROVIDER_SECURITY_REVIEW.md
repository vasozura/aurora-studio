## Threat Model

### T1 — Accidental Real Execution
**Risk**: Mock image adapter accidentally triggers a real network call.
**Mitigations**:
- `ImageProviderExecutionGate.evaluate_real_image()` always returns `allowed=False` in v0.4.
- `ImageProviderAdapter.execute_real_image()` is explicitly blocked and returns `status="blocked"`.
- `is_real_execution_allowed()` is hardcoded `False`.
- Test: `test_execute_real_blocked` in `test_mock_image_provider_adapter.py`.

### T2 — Secret Exfiltration via Response Payload
**Risk**: A leaked API key or token appears in response JSON.
**Mitigations**:
- `ImageProviderResponse` contains no secret-holding fields.
- `sanitize_response_payload()` strips forbidden keys.
- `sanitize_error()` redacts sensitive patterns from error strings.
- `to_dict()` verified to exclude `api_key`, `secret`, `token`, `password`.
- Test: `test_response_no_secret_fields` in `test_mock_image_provider_adapter.py`.

### T3 — Binary Payload Injection (image bytes)
**Risk**: A caller passes `image_bytes` or `image_base64` in request parameters, attempting to bypass safety or extract media.
**Mitigations**:
- `FORBIDDEN_PARAMETER_KEYS` blocks: `image_bytes`, `image_base64`, `mask_base64`, `reference_image_base64`, `upload_file`, `file_path`, `asset_binary`, `api_key`, `token`, `secret`, `password`.
- `validate_image_provider_request()` rejects requests with forbidden keys.
- `ImageProviderRequest` has no binary-holding fields in its schema.
- Test: `test_validate_image_provider_parameters_forbidden_keys` in `test_image_provider_contracts.py`.

### T4 — Image File Creation
**Risk**: Adapter creates image files on disk (PNG, JPEG, etc.).
**Mitigations**:
- `execute_mock()` builds only an in-memory `ImageProviderResponse`.
- `image_uri` is always the `mock://image/<request_id>` URI scheme.
- No `open()`, `write()`, `PIL`, `cv2`, or `moviepy` is imported in source.
- Test: `test_no_image_file_created` in both adapter and bridge test files.

### T5 — Forbidden Library Import
**Risk**: A future diff adds PIL, cv2, httpx, or similar, accidentally enabling network or binary decode.
**Mitigations**:
- Source scan in QA test: `test_image_provider_no_forbidden_imports_in_source`.
- CI should fail immediately if any forbidden import appears in `src/aurora_studio/**/*.py`.

### T6 — Prompt Injection via Export Artifact
**Risk**: Export artifact ID contains shell metacharacters or injection vectors.
**Mitigations**:
- `build_image_prompt_from_export()` wraps artifact ID in a plain text template string.
- No shell exec, no file read, no subprocess. Pure string composition.
- Prompt text is truncated in response preview to 120 characters.

### T7 — Run History Secrets Leak
**Risk**: Provider run history persists secrets between sessions.
**Mitigations**:
- Run history stored in a plain Python list on `ImagePromptExportBridge` — ephemeral, cleared on restart.
- No run history is written to disk, JSON files, autosave files, logs, or backups.
- `_build_result()` explicitly excludes secrets from the result dict.

## Security Decisions (v0.4)

| Decision | Rationale |
|---|---|
| Real image execution blocked (hardcoded) | v0.4 is mock-only; no real provider SDK available, no key storage, no image review pipeline |
| No PIL/cv2/moviepy in source | Prevents accidental binary decode and large transitive dependency surface |
| Forbidden parameter keys enforced at validation | Defense-in-depth — callers cannot smuggle binary data through the `parameters` tuple |
| No persistent secrets | Ephemeral model: secrets passed only at call time and never stored |
| mock://image/ URI scheme | Unambiguously identifies mock results; cannot be confused with real image data |
